# Data Architecture Guide

## Overview
The project uses synthetic monthly banking and regional data stored in SQLite, plus optional processed files generated by test collectors.

## Main layers
- **Generation layer**: `generate_mock_data.py` creates demo data.
- **Storage layer**: `data/database/uz_banking_demo.sqlite`.
- **Analytics layer**: `src/analytics/` computes KPIs and snapshots.
- **Presentation layer**: `app.py` routes to page modules in `src/dashboard/pages/`.
- **Collector test layer**: `collect_cbu_bankstats_test.py` stores raw/processed artifacts in `data/raw/` and `data/processed/`.

## Dashboard modular layout
- `app.py` handles app setup, shared filters, data loading, and tab routing.
- `src/dashboard/pages/` contains one file per tab to reduce merge conflicts.

## Key paths
- `data/database/` — SQLite database
- `data/raw/` — downloaded raw files
- `data/processed/` — parsed CSV outputs
- `src/common/` — constants and DB helpers
- `src/dashboard/charts.py` — reusable plotting helpers
- `src/dashboard/pages/` — tab/page render modules

## Extending safely
When adding new dashboard features:
1. Add a new page module under `src/dashboard/pages/`.
2. Add one new tab label and one `render(...)` call in `app.py`.
3. Keep shared filter logic in `app.py`, page-specific logic in page modules.
4. Avoid editing unrelated core files.
